## 🏛️ Architecture Highlights

Notes on decisions made while building GK CRM — the parts that don't show up in a screenshot but matter for a real multi-tenant product:

- **Row Level Security (RLS) as the source of truth** — tenant isolation and role-based visibility are enforced at the database layer, not just in application code, so a bug in the frontend can't leak another tenant's data
- **Claim-based lead ownership** — a Postgres trigger assigns lead ownership on first action, keeping `owner_id` (used by RLS) and `assigned_to` (used by the UI) in sync
- **Paginated data fetching** — custom fetch utilities bypass Supabase's default row caps for tenants with large datasets
- **Webhook-driven subscription state** — Razorpay webhooks are the single source of truth for plan/subscription activation, not client-side confirmation
- **Separation of concerns** — clear boundary between tenant-facing modules (CRM, HRMS, Billing) and the super-admin layer (tenant management, revenue dashboard, support tickets)
